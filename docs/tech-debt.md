## Архитектура и контракты

### P1 — Расширить error contract полем `data`

Сейчас формат ошибки: `{"error": {"code", "message"}}`. Когда клиенту нужны
структурированные данные (количество оставшихся попыток SMS-кода,
секунды до сброса rate limit), мы вставляем их в `message` строкой.

Это работает для отображения, но не машинно-читаемо.

**Решение**: расширить формат до `{"error": {"code", "message", "data"}}`,
где `data` — словарь с дополнительными полями. Обновить
`backend/apps/common/exception_handler.py` и базовый класс `AppError`
(дополнительный конструктор-параметр `data: dict | None`).

**Когда**: при добавлении Flutter-клиента, если потребуется парсить
remaining_attempts/retry_after из ответа.

### P1 — Завести `ServiceUnavailableError`

Сейчас в `backend/apps/users/api/v1/views.py:SmsRequestView` два места
отдают 503 через `return Response({"detail": ...}, status=503)` напрямую —
это нарушение единого контракта (см. `# TODO` в коде).

**Решение**: добавить в `backend/apps/common/exceptions.py` класс
`ServiceUnavailableError` (HTTP 503), добавить маппинг в
`_DOMAIN_ERROR_STATUSES`, переписать оба return на `raise`.

**Когда**: одновременно с пунктом про расширение error contract data
(или раньше, если нужно).

### P2 — `SmsCodeService` (сервисный слой)

SMS-функции из `backend/apps/users/api/v1/utils.py` (`save_sms_code`,
`get_sms_code`, `increment_attempts`, и т.д.) — это набор свободных
функций над Redis.

По гайдлайнам `clean_code.md` правило про сервисный слой — это должен быть
класс-сервис `SmsCodeService` с публичными методами и приватным состоянием.

**Когда**: при добавлении реального SMS-провайдера (вместо
`ConsoleSmsProvider`), потому что тогда сервис будет логичной точкой
инъекции зависимости.

## Внешние интеграции и релиз

### P0 — Apple Developer аккаунт: Individual vs Organization

ИП заказчика с D-U-N-S зарегистрирован как Individual. Имя издателя в
App Store будет «Юрий Радченко», не «Барахолка». Переход на Organization
для ИП проблематичен (Apple требует юр.лицо, ИП может не подойти).

**Решение нужно**: обсудить с заказчиком перед публичным релизом.

**Когда**: ближе к Этапу 5 (релиз), но не позже TestFlight-беты.

### P1 — Кириллический домен Барахолка.online (IDN)

Punycode: `xn--80aaab1cfuo0c.online`.

**Риск**: HTTP-клиенты в Flutter могут не работать с punycode-доменами
из коробки.

**Когда**: перед стартом Flutter — провести тест запроса с punycode-доменом.
Если не работает — обсудить с заказчиком регистрацию параллельного
латинского технического домена.

### P2 — Геокодинг по адресу в админке складов

Ветка `feature/map-address-search` остановлена: Nominatim банит публичные
запросы. Нужен бэкенд-прокси + платный геокодер (например Yandex
Геокодер).

**Когда**: при необходимости массового завода складов с адресами.

### P3 — drf-spectacular: ~14 W001/W002 предупреждений

Косметика типов полей в Swagger UI. Не влияет на функционал.

**Когда**: при полировке API-документации.

### P1 — `ConsoleSmsProvider`: print → logger

Сейчас 3 ошибки T201 в `backend/apps/notifications/sms/console.py`.
Заменить `print` на `logger`, **маскировать SMS-код в логах**
(по правилу безопасности «не логировать SMS-коды»). В тестах —
заглушить логгер.

### P2 — `manage.py` TRY003

Длинное сообщение в `ImportError` — это Django boilerplate, лучше не
менять. Добавить `manage.py` в `per-file-ignores` ruff с пояснением
комментарием в `backend/pyproject.toml`.

### P2 — `catalog/models.py:97` PLR2004

Магическое `2` — это максимум глубины категорий. Вынести в константу
модуля `MAX_CATEGORY_DEPTH = 3`, использовать `>= MAX_CATEGORY_DEPTH - 1`
в проверке. Перепроверить логику глубины при вынесении.
