## Архитектура и контракты

### ЗАКРЫТО (2026-07) — Расширить error contract полем деталей

Реализовано как поле `details` (не `data`) в Order-фазе. Формат:
`{"error": {"code", "message", "details"}}`, где `details` опционально
(dict или list). Используется в not_enough_stock (список товаров),
product_unavailable, cannot_cancel. `AppError` принимает `details`,
`exception_handler` включает его в JSON только если передан.

### P1 — Завести `ServiceUnavailableError`

Сейчас в `backend/apps/users/api/v1/views.py:SmsRequestView` два места
отдают 503 через `return Response({"detail": ...}, status=503)` напрямую —
это нарушение единого контракта (см. `# TODO` в коде).

**Решение**: добавить в `backend/apps/common/exceptions.py` класс
`ServiceUnavailableError` (HTTP 503), добавить маппинг в
`_DOMAIN_ERROR_STATUSES`, переписать оба return на `raise`.

**Когда**: одновременно с пунктом про расширение error contract data
(или раньше, если нужно).

### P2 — `SmsCodeService` (сервисный слой)

SMS-функции из `backend/apps/users/api/v1/utils.py` (`save_sms_code`,
`get_sms_code`, `increment_attempts`, и т.д.) — это набор свободных
функций над Redis.

По гайдлайнам `clean_code.md` правило про сервисный слой — это должен быть
класс-сервис `SmsCodeService` с публичными методами и приватным состоянием.

**Когда**: при добавлении реального SMS-провайдера (вместо
`ConsoleSmsProvider`), потому что тогда сервис будет логичной точкой
инъекции зависимости.

### P0 — Apple Developer аккаунт: Individual vs Organization

ИП заказчика с D-U-N-S зарегистрирован как Individual. Имя издателя в
App Store будет «Юрий Радченко», не «Барахолка». Переход на Organization
для ИП проблематичен (Apple требует юр.лицо, ИП может не подойти).

**Решение нужно**: обсудить с заказчиком перед публичным релизом.

**Когда**: ближе к Этапу 5 (релиз), но не позже TestFlight-беты.

### P1 — Кириллический домен Барахолка.online (IDN)

Punycode: `xn--80aaab1cfuo0c.online`.

**Риск**: HTTP-клиенты в Flutter могут не работать с punycode-доменами
из коробки.

**Когда**: перед стартом Flutter — провести тест запроса с punycode-доменом.
Если не работает — обсудить с заказчиком регистрацию параллельного
латинского технического домена.

### P2 — Геокодинг по адресу в админке складов

Ветка `feature/map-address-search` остановлена: Nominatim банит публичные
запросы. Нужен бэкенд-прокси + платный геокодер (например Yandex
Геокодер).

**Когда**: при необходимости массового завода складов с адресами.

## Заказы и оплата

### P1 — Расчёт стоимости доставки

Сейчас `delivery_cost` всегда `Decimal("0.00")` в CheckoutService.
Реальная стоимость зависит от способа доставки, расстояния до склада,
веса/габаритов. Нужен калькулятор доставки.

**Когда**: когда заказчик определит тарифы доставки.

### P1 — asap / scheduled доставка

Сейчас доставка без выбора времени. Нужна поддержка «как можно скорее»
vs «на конкретную дату/интервал». Требует полей желаемого времени в Order
и логики проверки доступных слотов.

**Когда**: по требованию заказчика для мобильного приложения.

### P1 — Webhook эквайринга вместо mark-paid

Сейчас оплата отмечается вручную через `POST /orders/{uuid}/mark-paid/`
(только is_staff) — заглушка. В проде нужен webhook платёжного провайдера,
который автоматически вызывает `PaymentStatusService.change_payment_status`
при поступлении оплаты.

**Когда**: при подключении реального эквайринга (ЮKassa/Тинькофф/др.).

### P1 — Ограничение «не отгружать неоплаченное»

По решению с Александром (две оси статусов) физический статус и оплата
независимы. Но бизнес-правило «нельзя shipped пока payment_status != paid»
для card_online нужно накинуть поверх — в OrderStatusService или отдельной
проверкой перед переходом в shipped.

**Когда**: перед запуском реальных продаж.

### P2 — Роли курьеров вместо is_staff для mark-paid

Сейчас mark-paid доступен любому is_staff. Когда появится доставка
курьерами (оплата при получении, если вернём cash_on_delivery), нужна
отдельная группа/роль «курьер» с правом отмечать оплату только по своим
заказам.

**Когда**: если введём наличную оплату и курьерскую доставку.

### P2 — PostgreSQL SEQUENCE для номеров заказов

Сейчас номер генерируется через `SELECT FOR UPDATE` + COUNT в
`CheckoutService._generate_order_number`. Для трёх ИП с десятками заказов
в день этого достаточно. При росте нагрузки — заменить на PostgreSQL
SEQUENCE per seller (быстрее, без блокировки строки Seller).

**Когда**: если число заказов вырастет до сотен в минуту.

### P2 — Смешанные заказы (stock + made_to_order)

`Order._get_product_type_for_transitions` использует made_to_order-граф
если хоть один товар made_to_order. Для MVP заказы обычно однотипны.
Уточнить логику если появятся регулярные смешанные заказы (например,
раздельные подзаказы или отдельные цепочки статусов).

**Когда**: если смешанные заказы станут частым сценарием.

### P2 — Индекс на Order.status для авто-отмены

Для будущей Celery-задачи «отменять неоплаченные заказы старше N часов»
понадобится индекс на `status` (и, возможно, составной с `created_at`),
чтобы periodic-задача не сканировала всю таблицу.

**Когда**: при внедрении Celery и авто-отмены зависших заказов.

## Checkout и оплата

### P1 — Поиск адреса по строке (геокодинг) на checkout

Заказчик хочет: пользователь вводит адрес строкой → на карте
подставляется точка. Это геокодинг (текст → координаты). Связано с
уже известной проблемой (см. «Геокодинг по адресу в админке складов»):
Nominatim банит публичные запросы, нужен платный Yandex Геокодер +
бэкенд-прокси. На старте checkout использует ТОЛЬКО ручную установку
точки на карте (координаты берутся с карты напрямую, бесплатно).

**Требует решения заказчика**: готов ли оплачивать Yandex Геокодер
ради поиска адреса строкой. Пока — карта с ручной точкой.

**Когда**: после согласования расходов на геокодер с заказчиком.

### P2 — cash_on_delivery оставлен в backend, скрыт в UI

По решению (2026-07) оплата на checkout — ТОЛЬКО card_online.
PaymentMethod.CASH_ON_DELIVERY остаётся в модели (не удаляем — лишняя
миграция, может вернуться), но в мобильном UI не предлагается.
Двухосевая архитектура статусов (physical status + payment_status)
сохраняется — она правильна независимо от способа оплаты (refunded
может случиться в любой момент).

**Когда**: если заказчик решит вернуть оплату при получении —
достаточно показать вариант в UI, backend уже готов.
