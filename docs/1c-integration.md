# Интеграция с 1С

Документ описывает согласованную архитектуру обмена данными между приложением Baraxolka и системой 1С: Управление торговлей 11.5.

Согласовано: Константин Денисов (разработчик приложения) и 1С-специалист заказчика.
Дата согласования: июнь 2026.

## 1. Общие принципы

### 1.1. Направление обмена

Инициатор всех запросов — **1С**. 1С приходит к нашему бэкенду:

- забирает данные о новых заказах
- передаёт обновления товаров, остатков и цен

Приложение **не отправляет** запросов в 1С самостоятельно. Это связано с тем, что 1С работает локально у заказчика и может быть недоступна из интернета постоянно.

### 1.2. Инфраструктура

- 1С: Управление торговлей 11.5
- Развёрнута локально (на стороне заказчика)
- Одна общая база данных на трёх ИП
- Доступ к копии 1С для разработки и отладки предоставляется

### 1.3. Технический протокол

- Формат данных: JSON
- Кодировка: UTF-8
- Транспорт: HTTPS (только)
- Идентификаторы объектов: UUID

## 2. Аутентификация

### 2.1. Принцип

Используется единый JWT-токен на весь обмен. Один токен покрывает все эндпоинты обмена, кроме самих эндпоинтов получения и обновления токена.

Реализация: OAuth 2.0 Client Credentials Grant.

### 2.2. Эндпоинты аутентификации

**Получение токена:**

\`\`\`
POST /api/1c/auth/token/
\`\`\`

Запрос:
\`\`\`json
{
  "client_id": "1c-integration",
  "client_secret": "<секрет>"
}
\`\`\`

Ответ:
\`\`\`json
{
  "access_token": "<JWT>",
  "refresh_token": "<refresh>",
  "expires_in": 3600,
  "token_type": "Bearer"
}
\`\`\`

**Обновление токена:**

\`\`\`
POST /api/1c/auth/refresh/
\`\`\`

Запрос:
\`\`\`json
{
  "refresh_token": "<refresh>"
}
\`\`\`

Ответ: аналогичен получению токена.

### 2.3. Использование токена

Во всех запросах от 1С передаётся заголовок:

\`\`\`
Authorization: Bearer <access_token>
\`\`\`

Время жизни access_token: 1 час.
Время жизни refresh_token: 30 дней.

## 3. Передача данных из 1С в приложение

### 3.1. Товары

1С передаёт справочник товаров в приложение. Учёт ведётся в 1С, приложение — потребитель данных.

**Поля товара:**

- `uuid` — UUID товара в 1С (уникальный идентификатор)
- `name_short` — наименование краткое (для каталога)
- `name_full` — наименование полное (для карточки товара)
- `description` — описание (опционально)
- `seller_uuid` — UUID продавца, владельца товара
- `price` — цена, тип decimal с двумя знаками после запятой (рубли)
- `is_available_for_sale` — флаг доступности к продаже (если false — товар не отображается в каталоге даже при положительном остатке)
- `stocks` — список остатков по складам (см. ниже)

**Остатки в разрезе складов:**

\`\`\`json
"stocks": [
  {
    "warehouse_uuid": "<UUID склада>",
    "quantity": 5
  }
]
\`\`\`

Передаётся итоговый остаток к продаже — уже за вычетом брака.

### 3.2. Учёт брака

Брак ведётся на стороне 1С на отдельных складах с типом «Брак». Эти склады в выгрузке не участвуют.

Итог: приложение видит только остаток, который реально можно продать.

### 3.3. Цены и их изменения

При синхронизации обновлений цен 1С передаёт по каждому изменённому товару:

- `uuid` — UUID товара
- `changed_at` — время изменения в формате unixtime (целое число секунд)
- `new_price` — новая цена, decimal

Валюта не передаётся (всегда рубли).

### 3.4. Эндпоинт приёма данных

Конкретный формат вебхуков/эндпоинтов на стороне приложения будет согласован отдельно при реализации (Этап 4 — интеграция с 1С).

## 4. Передача заказов из приложения в 1С

### 4.1. Принцип

1С периодически опрашивает наш бэкенд и забирает новые/изменённые заказы. После создания документа отгрузки в 1С — присылает подтверждение.

### 4.2. Эндпоинт получения заказов

\`\`\`
GET /api/1c/orders/?since=<timestamp>
\`\`\`

Параметр `since` — unixtime последней успешной синхронизации. Возвращаются заказы, изменённые после этого момента.

Ответ:

\`\`\`json
{
  "orders": [
    {
      "uuid": "<UUID заказа>",
      "created_at": 1735689600,
      "status": "shipped",
      "payment_method": "card_online",
      "client_id": "<UUID клиента>",
      "warehouse_uuid": "<UUID склада отгрузки>",
      "delivery_method": "courier",
      "total_amount": "1290.50",
      "comment": "Звонить заранее",
      "items": [
        {
          "product_uuid": "<UUID товара>",
          "seller_uuid": "<UUID продавца>",
          "quantity": 2,
          "price": "645.25",
          "sum": "1290.50"
        }
      ]
    }
  ]
}
\`\`\`

### 4.3. Эндпоинт подтверждения

После создания документа в 1С:

\`\`\`
POST /api/1c/orders/{order_uuid}/confirm/
\`\`\`

Запрос:

\`\`\`json
{
  "success": true,
  "docnum": 12345,
  "errorcode": null
}
\`\`\`

Если 1С не смогла создать документ:

\`\`\`json
{
  "success": false,
  "docnum": null,
  "errorcode": "STOCK_MISMATCH"
}
\`\`\`

Приложение сохраняет результат у каждого заказа.

## 5. Склады

### 5.1. Где что ведётся

| Что | Где ведётся |
|---|---|
| UUID склада | 1С (источник истины) |
| Связь склад → продавец | 1С |
| Координаты центра склада | Приложение (админка) |
| Полигон зоны доставки | Приложение (админка, через карту) |
| Флаг доступности самовывоза | Приложение (админка) |

### 5.2. Логика выбора склада при оформлении заказа

**Сценарий «Самовывоз»:**
1. Покупатель выбирает «забрать самому»
2. Видит список складов с самовывозом или карту
3. Выбирает удобный — заказ привязывается к этому складу

**Сценарий «Доставка»:**
1. Покупатель вводит адрес доставки
2. Система определяет, в какую зону доставки попадает адрес
3. Если адрес попадает в одну зону — заказ привязывается к складу этой зоны
4. Если адрес попадает в несколько зон (кроссы между складами разных ИП) — покупатель выбирает один из вариантов с указанием расчётного времени доставки

### 5.3. Кроссы зон доставки

Допускается, что зоны доставки нескольких складов могут пересекаться. Например, в одном районе города могут работать склады трёх разных ИП. В этом случае покупателю предлагается выбор.

## 6. Расширения исходного ТЗ

В процессе согласования выявлены следующие расширения объёма работ относительно исходного ТЗ:

### 6.1. Цены из 1С

Исходно: цены ведутся только в админке приложения.
Финал: базовая цена приходит из 1С, в админке можно ставить скидки/акции.

### 6.2. Передача заказов в 1С

Исходно: заказы только в админке приложения, в 1С не передаются.
Финал: реализована двусторонняя интеграция — 1С забирает заказы и подтверждает создание документов отгрузки.

### 6.3. Мультискладовая модель с зонами доставки

Исходно: простая модель «склад продавца».
Финал: каждый склад имеет полигон зоны доставки на карте, выбор склада зависит от адреса клиента.

### 6.4. Оформление расширений

Все расширения подлежат фиксации в дополнительном соглашении к договору. Допсоглашение готовится ближе к Этапу 4 (интеграция с 1С), когда полный объём интеграционных работ становится понятным для оценки.

## 7. Сценарии ошибок

Сценарии обработки ошибок (товара нет в 1С, остатков не хватает, 1С недоступна и т.д.) подлежат отдельной проработке на этапе реализации интеграции. Будет проведён созвон с 1С-специалистом и составлен отдельный документ.
