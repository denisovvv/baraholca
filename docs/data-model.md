# Модели данных

Документ описывает все модели данных приложения Baraxolka. Это спецификация, на основе которой пишется код.

## Общие соглашения

### Идентификаторы

Все модели имеют поле `id` (BigAutoField, автоинкрементное) — внутренний идентификатор Django.

Модели, синхронизируемые с 1С, дополнительно имеют поле `uuid_1c` (UUIDField) — идентификатор объекта в 1С. По нему происходит сопоставление при синхронизации.

Внешние API (мобильное приложение, веб-админка) работают с внутренним `id`, не с UUID 1С.

### Даты

Каждая модель имеет два поля времени:

- `created_at` — DateTimeField, auto_now_add=True (заполняется при создании, не меняется)
- `updated_at` — DateTimeField, auto_now=True (обновляется при каждом сохранении)

Часовой пояс приложения: Europe/Moscow.

### Удаление

Мы НЕ удаляем большинство объектов физически. Используем флаг `is_active` (BooleanField, default=True). Если флаг выключен — объект скрыт от пользователей, но остаётся в базе для истории.

Физическое удаление применяется только к временным объектам: SmsCode, CartItem.

### Денежные значения

Все денежные поля — DecimalField(max_digits=12, decimal_places=2).

Это позволяет хранить значения до 9 999 999 999.99 ₽ с точностью до копеек.

### Связь с пользователем

Внешние ключи на User используют `settings.AUTH_USER_MODEL`, не прямой импорт. Это стандарт Django.

## Слой 1: Пользователи и продавцы

### User

Кастомная модель пользователя. Авторизация по номеру телефона. Используется для покупателей, администраторов продавцов и суперпользователей системы.

**Расположение:** `apps/users/models.py`

**Поля:**

**Поля:**

| Поле | Тип | Описание |
|---|---|---|
| `id` | BigAutoField | Первичный ключ |
| `phone` | CharField(20), unique, null, blank | Номер телефона в формате `+7XXXXXXXXXX`. Не обязателен на уровне БД, но обязателен на уровне бизнес-логики (нельзя оформить заказ без подтверждённого телефона) |
| `phone_verified` | BooleanField, default=False | Подтверждён ли телефон (через flash call или SMS) |
| `first_name` | CharField(150), blank | Имя |
| `last_name` | CharField(150), blank | Фамилия |
| `email` | EmailField, blank | Email (опционально, может приходить из соцсети) |
| `vk_id` | CharField(50), unique, null, blank | ID пользователя в VK (для входа через VK ID) |
| `apple_id` | CharField(255), unique, null, blank | ID пользователя в Apple (sub-токен из Apple Sign In) |
| `social_avatar_url` | URLField(500), blank | Аватар из соцсети (при первом входе) |
| `is_active` | BooleanField, default=True | Активен (можно входить) |
| `is_staff` | BooleanField, default=False | Доступ в админ-панель |
| `is_superuser` | BooleanField, default=False | Суперпользователь (через PermissionsMixin) |
| `date_joined` | DateTimeField | Дата регистрации |
| `last_login` | DateTimeField | Дата последнего входа (стандартное поле Django) |


**Особенности:**

- `USERNAME_FIELD = 'phone'` — но phone может быть пустым при первом входе через соцсеть
- Один из идентификаторов (phone, vk_id, apple_id) обязателен — это контролируется на уровне валидации модели
- Покупатели не имеют пароля (входят по коду подтверждения через flash call/SMS, либо через соцсеть)
- Продавцы и суперпользователи имеют пароль для входа в админ-панель
- Пароль (если есть) хранится в захэшированном виде в стандартном поле `password`
- `phone_verified=True` — обязательное условие для оформления заказа
- `vk_id` и `apple_id` уникальны — один аккаунт VK / Apple может быть привязан только к одному User

**Менеджер:**

`UserManager` с методами `create_user(phone, password=None)` и `create_superuser(phone, password)`.

### SellerStaff

Связующая модель: какие пользователи работают у какого продавца и с какой ролью.

Нужна, потому что у одного продавца может быть несколько сотрудников с разными правами (главный администратор, менеджер заказов, кладовщик и т.д.), и один пользователь теоретически может работать у нескольких продавцов.

**Расположение:** `apps/sellers/models.py`

**Поля:**

| Поле | Тип | Описание |
|---|---|---|
| `id` | BigAutoField | Первичный ключ |
| `user` | ForeignKey → User, CASCADE | Пользователь |
| `seller` | ForeignKey → Seller, CASCADE | Продавец |
| `role` | CharField(20), choices | Роль: 'admin', 'manager', 'storekeeper' |
| `is_active` | BooleanField, default=True | Активен |
| `created_at` | DateTimeField | Создан |

**Уникальный индекс:** `(user, seller)` — один пользователь у одного продавца имеет только одну активную роль.

**Роли и права:**

- `admin` — полный доступ к данным продавца: товары, заказы, склады, статистика
- `manager` — управление заказами и статусами, без доступа к редактированию товаров
- `storekeeper` — только просмотр и обработка заказов на конкретном складе

Конкретные права прописываются в админке через Permissions Django.

## Социальная авторизация

Дополнительный документ описывает логику входа через VK и Apple.

### Поддерживаемые провайдеры

На старте проекта подключены:

- **VK ID** — основная российская соцсеть, релевантна аудитории заказчика (~30 000 подписчиков в группе)
- **Apple Sign In** — обязателен при наличии других соцсетей (требование App Store)

Не подключены:

- Google Sign In — не нужен на старте, может быть добавлен позже
- MAX — SDK сыроват, добавим во второй версии
- Email с паролем — отказались как от устаревшего

### Логика входа через соцсеть

**Сценарий «Первый вход через VK или Apple»:**

1. Пользователь нажимает «Войти через VK» (или Apple)
2. Открывается окно провайдера, пользователь авторизуется и подтверждает доступ
3. Мобильное приложение получает токен от провайдера
4. Приложение отправляет токен на наш бэкенд: `POST /api/auth/social/vk/` (или `/apple/`)
5. Бэкенд проверяет токен через API провайдера, получает идентификатор и базовые данные (имя, фамилия, аватар, email если разрешено)
6. Бэкенд ищет существующего User с этим vk_id (или apple_id):
   - Если нет — создаёт нового User с заполнением vk_id/apple_id, имени, фамилии, аватара
   - Если есть — использует существующего
7. Бэкенд проверяет, заполнен ли phone и phone_verified=True
   - Если да — выдаёт JWT-токены, пользователь входит в приложение
   - Если нет — выдаёт временный токен и направляет пользователя на экран «Подтвердите телефон»
8. На экране подтверждения телефона пользователь вводит номер
9. Запускается flash call/SMS, как для обычного входа по телефону
10. После подтверждения phone_verified=True, выдаются полноценные JWT-токены, вход в приложение

**Сценарий «Повторный вход через VK/Apple»:**

Шаги 1-5 такие же. На шаге 6 находится существующий User с подтверждённым phone — сразу выдаются JWT-токены.

**Сценарий «Привязка соцсети к существующему аккаунту»:**

Если пользователь ранее вошёл по телефону, а потом нажал «Войти через VK» с того же устройства — мы можем привязать VK к существующему аккаунту. Логика:

1. Пользователь уже залогинен (через phone)
2. В настройках профиля нажимает «Привязать VK»
3. Происходит вход через VK, мы получаем vk_id
4. Проверяем: нет ли уже другого User с таким vk_id?
   - Если есть — ошибка «этот VK уже привязан к другому аккаунту»
   - Если нет — записываем vk_id в текущего User

### Эндпоинты для социальной авторизации

POST /api/auth/social/vk/
POST /api/auth/social/apple/
POST /api/auth/phone-verify/   (для подтверждения телефона после соцвхода)
POST /api/auth/social/link/    (привязка соцсети к существующему аккаунту)
POST /api/auth/social/unlink/  (отвязка соцсети)

Конкретные форматы запросов и ответов — в документе `api.md` (будет создан отдельно).

### Безопасность

- Токены от провайдеров **всегда** проверяются на сервере (никогда не доверяем тому, что прислал клиент без проверки)
- vk_id и apple_id уникальны на уровне БД — нельзя привязать один аккаунт соцсети к двум разным User
- При смене номера телефона пользователь должен подтвердить новый через flash call
- Логи попыток входа через соцсети ведутся в отдельной модели SocialAuthLog (для безопасности и отладки)

### Warehouse

Склад / точка продавца. У одного продавца может быть несколько складов (например, разные точки в городе).

UUID склада приходит из 1С. Координаты и зона доставки ведутся в админке приложения.

**Расположение:** `apps/catalog/models.py`

**Поля:**

| Поле | Тип | Описание |
|---|---|---|
| `id` | BigAutoField | Первичный ключ |
| `uuid_1c` | UUIDField, unique | UUID склада в 1С |
| `seller` | ForeignKey → Seller, CASCADE | Продавец, которому принадлежит склад |
| `name` | CharField(150) | Название склада (например, «Склад на Ленина, 5») |
| `address` | CharField(500) | Адрес склада |
| `location` | PointField (PostGIS), SRID 4326 | Координаты центра склада (широта/долгота как одна точка) |
| `delivery_area` | PolygonField (PostGIS), SRID 4326, null | Полигон зоны доставки (необязателен — можно нарисовать позже) |
| `pickup_available` | BooleanField, default=True | Доступен ли самовывоз с этого склада |
| `working_hours` | JSONField, blank | Режим работы по дням недели |
| `contact_phone` | CharField(20), blank | Телефон склада (для покупателей при самовывозе) |
| `is_active` | BooleanField, default=True | Активен |
| `created_at` | DateTimeField | Создан |
| `updated_at` | DateTimeField | Обновлён |

**Особенности:**

- Поля `location` и `delivery_area` требуют расширения PostGIS в PostgreSQL
- На локальной разработке: установлен PostgreSQL 17 + PostGIS 3.6 через Homebrew
- На Yandex Cloud: PostGIS включается флагом при создании managed PostgreSQL
- Используется SRID 4326 (WGS 84) — мировой стандарт географических координат (как у GPS, Google Maps, Яндекс.Карт)
- В админке для редактирования полей используется встроенная `django.contrib.gis.admin.GISModelAdmin` с картой OpenStreetMap
- Карта в админке центрируется на регионе Воронежская/Белгородская область
- В админке зона доставки рисуется на карте через JavaScript-библиотеку (Leaflet.draw или аналог)
- При оформлении заказа адрес клиента проверяется на попадание в `delivery_area` каждого склада

**Пример `working_hours`:**

```json
{
  "monday": {"open": "09:00", "close": "21:00"},
  "tuesday": {"open": "09:00", "close": "21:00"},
  "sunday": null
}
```

`null` означает выходной.

### Логика выбора склада при оформлении заказа

При оформлении заказа покупатель указывает адрес доставки. Система:

1. Находит все склады, в зону доставки которых попадает адрес клиента
2. Фильтрует склады по наличию товара (для каждого товара в корзине должен быть остаток на выбранном складе)
3. Если подходящий склад один — выбирается автоматически
4. Если складов несколько — покупатель выбирает (с указанием расчётного времени доставки для каждого)
5. Если ни один не подходит — сообщение «Доставка по вашему адресу временно недоступна»

Аналогичная логика для самовывоза, но без проверки попадания в зону.

### Product

Товар. Главная модель каталога.

Большая часть данных приходит из 1С (название, базовая цена, остатки). В админке приложения управляются: фото, скидки, флаг доступности, тип товара.

**Расположение:** `apps/catalog/models.py`

**Поля:**

| Поле | Тип | Описание |
|---|---|---|
| `id` | BigAutoField | Первичный ключ |
| `uuid_1c` | UUIDField, unique | UUID товара в 1С |
| `name_short` | CharField(255) | Краткое название (для каталога, списков) |
| `name_full` | CharField(500) | Полное название (для карточки товара) |
| `description` | TextField, blank | Описание |
| `seller` | ForeignKey → Seller, PROTECT | Продавец |
| `category` | ForeignKey → Category, PROTECT, null | Категория |
| `product_type` | CharField(20), choices | Тип: 'stock' (со склада) или 'made_to_order' (под заказ) |
| `base_price` | DecimalField(12,2) | Базовая цена из 1С (рубли) |
| `discount_price` | DecimalField(12,2), null, blank | Скидочная цена (если задана, используется вместо base_price) |
| `is_available_for_sale` | BooleanField | Флаг доступности (приходит из 1С) |
| `production_time_days` | IntegerField, null, blank | Время изготовления в днях (только для made_to_order) |
| `is_active` | BooleanField, default=True | Активен в нашей системе |
| `created_at` | DateTimeField | Создан |
| `updated_at` | DateTimeField | Обновлён |
| `synced_at` | DateTimeField, null | Время последней синхронизации с 1С |

**Особенности:**

- `seller` через PROTECT — нельзя удалить продавца, если у него есть товары
- `category` через PROTECT — нельзя удалить категорию, если в ней есть товары
- При расчёте цены для покупателя используется `discount_price` если он задан, иначе `base_price`. Метод `get_effective_price()` возвращает актуальную цену.
- `is_available_for_sale` управляется со стороны 1С — это «глобальный» выключатель товара
- `is_active` управляется в админке приложения — для случаев, когда нужно временно скрыть товар по нашей инициативе (например, проблемы с фото)
- Товар отображается в каталоге если: `is_available_for_sale = True`, `is_active = True`, `seller.is_active = True`, и есть положительный остаток хотя бы на одном складе (для stock-товаров) либо `production_time_days` задано (для made_to_order)

**Типы товара:**

- `stock` — товар со склада. Имеет остатки в `ProductStock`. При заказе списывается со склада.
- `made_to_order` — товар под заказ (печать на 3D-принтере). Не имеет остатков. При заказе появляется статус «в производстве», после изготовления — «готов».

- При загрузке автоматически создаются несколько вариантов: thumbnail (200x200), medium (600x600), large (1200x1200). Используется библиотека `django-imagekit` или аналог.
- У каждого товара должно быть ровно одно `is_main=True`. Если ставится новое — старое автоматически сбрасывается (логика в `save()` метода).

### ProductStock

Остатки товара в разрезе складов. Связующая модель между Product и Warehouse.

Приходит из 1С. Управляется только синхронизацией, в админке не редактируется вручную.

**Расположение:** `apps/catalog/models.py`

**Поля:**

| Поле | Тип | Описание |
|---|---|---|
| `id` | BigAutoField | Первичный ключ |
| `product` | ForeignKey → Product, CASCADE | Товар |
| `warehouse` | ForeignKey → Warehouse, CASCADE | Склад |
| `quantity` | IntegerField, default=0 | Доступное количество (уже за вычетом брака и резервов) |
| `reserved_quantity` | IntegerField, default=0 | Количество в активных резервах под заказы |
| `updated_at` | DateTimeField | Время последнего обновления |

**Уникальный индекс:** `(product, warehouse)` — для одного товара на одном складе одна запись.

**Особенности:**

- `quantity` приходит из 1С и означает «доступно к продаже» (брак уже вычтен на стороне 1С)
- `reserved_quantity` ведётся в приложении: при подтверждении заказа количество товара переходит из доступного в резерв до момента отгрузки
- Доступное к покупке = `quantity - reserved_quantity`
- Если для товара type=`stock` нет ни одной записи `ProductStock` с положительным остатком — товар не отображается в каталоге

---

### Расчёт «есть в наличии» для каталога

Когда покупатель смотрит каталог, для каждого товара рассчитывается:

has_stock = exists(
ProductStock where product=товар AND quantity - reserved_quantity > 0
)

Для товаров `made_to_order` поле всегда `True` (если активен).

Это вычисляется на лету или кэшируется отдельным полем (решение по производительности — на этапе оптимизации).

Корзина не привязана к продавцу — в одну корзину можно положить товары от разных продавцов. При оформлении заказа корзина автоматически разбивается на отдельные заказы по продавцам (см. Слой 4).

**Расположение:** `apps/cart/models.py`

**Поля:**

| Поле | Тип | Описание |
|---|---|---|
| `id` | BigAutoField | Первичный ключ |
| `user` | OneToOneField → User, CASCADE | Владелец корзины |
| `created_at` | DateTimeField | Создана |
| `updated_at` | DateTimeField | Обновлена |

**Особенности:**

- OneToOneField — у одного пользователя ровно одна корзина
- Корзина создаётся автоматически при первом добавлении товара (через сигнал или метод `get_or_create_cart()`)
- Корзина не удаляется при оформлении заказа — её содержимое (CartItem) очищается, сама корзина остаётся

---

### CartItem

Позиция в корзине. Конкретный товар с количеством.

**Расположение:** `apps/cart/models.py`

**Поля:**

| Поле | Тип | Описание |
|---|---|---|
| `id` | BigAutoField | Первичный ключ |
| `cart` | ForeignKey → Cart, CASCADE | Корзина |
| `product` | ForeignKey → Product, CASCADE | Товар |
| `quantity` | PositiveIntegerField | Количество |
| `added_at` | DateTimeField | Время добавления |

**Уникальный индекс:** `(cart, product)` — в одной корзине один товар представлен одной строкой (увеличиваем `quantity`, не плодим строки).

**Особенности:**

- В отличие от большинства моделей, CartItem удаляется физически при удалении из корзины (это временные данные)
- При добавлении в корзину товара, которого уже там нет — создаётся новая запись с `quantity=1` (или больше, если указано)
- При добавлении товара, который уже в корзине — `quantity` увеличивается
- При оформлении заказа все CartItem из корзины удаляются физически
- Корзина НЕ резервирует товары на складе. Резерв происходит только в момент оформления заказа.

---

### Резервирование товара

Это важный момент, который часто путают.

**Вариант резервирования при добавлении в корзину** — плохой: пользователи кладут товары и забывают, склад «занят» неделями, другие покупатели не могут купить.

**Вариант резервирования при оформлении заказа** — хороший: когда пользователь нажал «Оформить заказ» и ввёл данные доставки, мы резервируем товар на 15-30 минут, чтобы он успел оплатить.

**Наша схема:**

1. Покупатель кладёт товар в корзину — никакого резерва. Просто запись в CartItem.
2. Покупатель идёт в оформление заказа — перепроверяем актуальные остатки.
3. Создаём заказ со статусом «pending_payment». В этот момент `ProductStock.reserved_quantity` увеличивается на количество.
4. Покупатель оплачивает в течение 15 минут — заказ переходит в «paid». Резерв сохраняется до отгрузки.
5. Покупатель не оплатил за 15 минут — заказ автоматически отменяется (через Celery), `reserved_quantity` уменьшается.
6. После отгрузки `ProductStock.quantity` уменьшается на количество, `reserved_quantity` тоже уменьшается. Информация уходит в 1С.

| Поле | Тип | Описание |

---

### Статусы заказа

Статус заказа меняется по мере его обработки. Цепочки разные для товаров со склада и под заказ.

**Для товаров типа `stock`:**

pending_payment → paid → assembling → shipped → in_delivery → delivered
↓
cancelled (отмена возможна на любом этапе до shipped)

**Для товаров типа `made_to_order` (3D-печать):**

pending_payment → paid → in_production → produced → assembling → shipped → in_delivery → delivered
↓
cancelled (отмена возможна до начала производства)

**Описание статусов:**

| Статус | Описание | Кто меняет |
|---|---|---|
| `pending_payment` | Заказ создан, ожидает оплаты | Система при создании |
| `paid` | Оплачен, готов к обработке | Платёжный сервис (вебхук) |
| `in_production` | В производстве (3D-печать) | Продавец вручную |
| `produced` | Изготовлен | Продавец вручную |
| `assembling` | Собирается на складе | Продавец вручную |
| `shipped` | Передан курьеру | Продавец вручную (триггер для 1С) |
| `in_delivery` | В пути к покупателю | Продавец вручную или автоматически по таймеру |
| `delivered` | Доставлен | Продавец вручную или подтверждение покупателя |
| `cancelled` | Отменён | Покупатель (до paid) или продавец (с обоснованием) |

**При переходе в `shipped`:**
- `ProductStock.quantity` уменьшается на количество
- `ProductStock.reserved_quantity` уменьшается на количество
- Заказ становится доступен 1С через GET /api/1c/orders/?since=...
- После подтверждения 1С — заполняется `1c_docnum` и `1c_synced_at`

## Слой 5: Отзывы

### Review

Отзыв покупателя.

Архитектурный момент с заделом на будущее: модель спроектирована так, чтобы поддерживать **два типа отзывов** — на товар и на продавца. На старте делаем только отзыв на товар, но структура позволит легко добавить отзыв на продавца без переделок.

**Расположение:** `apps/reviews/models.py`

**Поля:**

| Поле | Тип | Описание |
|---|---|---|
| `id` | BigAutoField | Первичный ключ |
| `user` | ForeignKey → User, CASCADE | Автор отзыва |
| `target_type` | CharField(20), choices | Тип объекта: 'product' или 'seller' |
| `product` | ForeignKey → Product, CASCADE, null, blank | Товар (если target_type='product') |
| `seller` | ForeignKey → Seller, CASCADE, null, blank | Продавец (если target_type='seller') |
| `order` | ForeignKey → Order, SET_NULL, null | Связанный заказ (для проверки, что покупатель действительно купил) |
| `rating` | PositiveSmallIntegerField | Оценка 1-5 |
| `text` | TextField, blank | Текст отзыва |
| `is_published` | BooleanField, default=True | Опубликован (для модерации в будущем) |
| `created_at` | DateTimeField | Создан |
| `updated_at` | DateTimeField | Обновлён |

**Особенности и правила:**

- Поля `product` и `seller` — взаимоисключающие. Заполняется одно из двух, в зависимости от `target_type`
- Покупатель может оставить отзыв на товар только если у него есть `delivered` заказ с этим товаром
- Один пользователь — один отзыв на один товар. Уникальный индекс `(user, product)` при `target_type='product'`
- При повторной попытке отзыва — открывается форма редактирования существующего
- Отзыв на продавца — функционал второй версии. Сейчас структура есть, но эндпоинты на API не делаем.

**Расчёт рейтинга товара:**

`Product.average_rating` — вычисляется на лету или кэшируется отдельно:

average_rating = AVG(rating) WHERE product=X AND is_published=True
review_count   = COUNT(*) WHERE product=X AND is_published=True

Хранение в Product как кэш-поля или вычисление в сериализаторе — решение по производительности на этапе оптимизации.

## Миграционная стратегия

При первой реализации модели создаются в следующем порядке (чтобы не было проблем со связями):

1. User, UserManager (apps.users)
   - Первая миграция: phone-based аутентификация
   - Вторая миграция (после согласования): поля vk_id, apple_id, phone_verified, social_avatar_url
   - Третья миграция: модель SocialAuthLog
2. Seller, SellerStaff (apps.sellers)
3. Category, Warehouse (apps.catalog)
4. Product, ProductImage, ProductStock (apps.catalog)
5. Cart, CartItem, Favorite (apps.cart)
6. Order, OrderItem, OrderStatusHistory (apps.orders)
7. Review (apps.reviews)
8. SmsCode (apps.users — расширение)
9. Discount (apps.catalog — расширение)

Каждое приложение коммитится отдельным коммитом с осмысленным сообщением.
